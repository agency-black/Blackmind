### 🔴 HALLAZGO 001 — CRÍTICO: `sandbox: false` en WebContentsView del Auth Popup

**Ubicación:** `electron/main.js` · Líneas ~330–345 (bloque `authPopupView = new WebContentsView({...})`)

**Descripción técnica:**
El WebContentsView que carga URLs de terceros (popups de autenticación OAuth/SSO) es instanciado con `sandbox: false`. En Electron, esto significa que el proceso renderer de esa vista tiene acceso completo a las APIs de Node.js vía el preload script. Dado que este view carga URLs externas arbitrarias detectadas por palabras clave (login, oauth, authorize, etc.), cualquier inyección de script en esas páginas externas podría escalar a ejecución de código nativo. El preload script (`preload.js`) está igualmente compartido con esta vista, exponiendo `electronAPI` completo.

**Justificación:**  
OWASP Electron Security Guideline EL-01: Todos los renderers que cargan contenido remoto deben correr en sandbox. CWE-269: Improper Privilege Management.

**Prompt para corrección:**
```
En el archivo electron/main.js, dentro de la función `ensureAuthPopupWindow()`, 
localiza el bloque `authPopupView = new WebContentsView({ webPreferences: { ... } })`.
Cambia `sandbox: false` por `sandbox: true`.
Elimina también la línea `preload: path.join(__dirname, './preload.js')` de ese bloque, 
ya que un sandbox verdadero no necesita (ni debe) tener el mismo preload que expone electronAPI completo.
Si se requiere comunicación con el auth popup, crea un preload mínimo separado: 
`electron/preload-auth.js` que solo exponga los métodos estrictamente necesarios 
(authPopup.close, authPopup.minimize, authPopup.toggleMaximize, authPopup.onState, authPopup.onCompleted).
```

### 🔴 HALLAZGO 002 — CRÍTICO: `setWindowOpenHandler` en `app.on('web-contents-created')` permite apertura sin restricciones

**Ubicación:** `electron/main.js` · Líneas ~1006–1008

```javascript
contents.setWindowOpenHandler((details) => {
  return { action: "allow" };
});
```

**Descripción técnica:**
El manejador global de `web-contents-created` establece `{ action: "allow" }` para **toda** ventana emergente, sin validar el protocolo, el dominio ni el origen. Esto significa que cualquier webview o contenido externo puede abrir ventanas arbitrarias con `window.open()`, incluyendo URLs con protocolo `javascript:`, `file://` local, o `blob:` data URIs con código malicioso. La falta de filtrado anula parcialmente el trabajo del `isTrustedOrigin()` implementado en otros módulos.

**Justificación:**
CVE-2023-4863 (similar patrón), Electron Security Recommendations §9: Use `setWindowOpenHandler` to validate URLs.

**Prompt para corrección:**
```
En electron/main.js, localiza el callback de `app.on('web-contents-created', ...)`.
Reemplaza el setWindowOpenHandler global por la siguiente implementación:

contents.setWindowOpenHandler(({ url, frameName, features }) => {
  try {
    const parsed = new URL(url);
    // Solo permitir http/https
    if (parsed.protocol !== 'http:' && parsed.protocol !== 'https:') {
      shell.openExternal(url).catch(() => {});
      return { action: 'deny' };
    }
    // Bloquear file:// y javascript:
    return { action: 'allow' };
  } catch {
    return { action: 'deny' };
  }
});

Esto previene que protocolos peligrosos (javascript:, file://, blob:) sean 
abiertos como nuevas ventanas de Electron.
```

### 🔴 HALLAZGO 003 — CRÍTICO: Descifrado sin autenticación — Fallback de encriptación en texto plano Base64

**Ubicación:** `electron/main.js` · Funciones `encryptString()` y `decryptString()` (líneas ~80–105)

**Descripción técnica:**
Cuando `safeStorage.isEncryptionAvailable()` retorna `false` (Linux sin keyring, VMs, entornos CI), las contraseñas se almacenan y recuperan en Base64 puro — que NO es encriptación, sino solo codificación. El archivo `passwords.json` en `userData` quedaría completamente legible. El sistema no advierte al usuario de este estado degradado ni bloquea el guardado de credenciales bajo este escenario.

**Justificación:**
CWE-311: Missing Encryption of Sensitive Data. CWE-326: Inadequate Encryption Strength.

**Prompt para corrección:**
```
En electron/main.js, modifica las funciones `encryptString` y `decryptString`:

1. En `encryptString`: Si `safeStorage.isEncryptionAvailable()` es false, 
   NO almacenes el valor. En su lugar retorna null y lanza un Error descriptivo:
   throw new Error('[Security] safeStorage no disponible. No se puede almacenar contraseña de forma segura.');

2. En `decryptString`: Si `safeStorage.isEncryptionAvailable()` es false,
   retorna null con un log de advertencia prominente.

3. En el handler `passwords:set`: Si `encryptString` retorna null o lanza error,
   retorna false al renderer y envía un evento IPC `security:storage-unavailable` 
   para que la UI informe al usuario.

4. Documenta en README.md que el gestor de contraseñas requiere un sistema 
   con keychain/keyring disponible.
```

### 🟠 HALLAZGO 004 — ALTO: `nodeIntegrationInWorker: true` activa Node.js en Web Workers

**Ubicación:** `electron/modules/WindowConfigurator.js` · Función `configureWebPreferences()` · Línea ~130

**Descripción técnica:**
La opción `nodeIntegrationInWorker: true` habilita las APIs de Node.js dentro de Web Workers del renderer. **Justificación:**
Electron Security Guideline §5: Do not enable `nodeIntegrationInWorker` unless strictly necessary. CWE-272: Least Privilege Violation.

**Prompt para corrección:**
```
En electron/modules/WindowConfigurator.js, en la función `configureWebPreferences()`,
elimina o cambia a false la línea:
  nodeIntegrationInWorker: true,

Si algún worker legítimo necesita funcionalidad de Node.js, usa en su lugar 
un Service Worker con comunicación IPC controlada a través del preload script,
o implementa el worker usando Electron's utility process (utilityProcess.fork())
que tiene aislamiento explícito.

Actualiza el comentario de Requisito 3.2 en el archivo para documentar la razón del cambio.
```

### 🟠 HALLAZGO 006 — ALTO: `downloadToPath` — Descarga sin validación de tipo MIME ni límite de tamaño

**Ubicación:** `electron/main.js` · Función `downloadToPath()` (líneas ~116–135)

**Descripción técnica:**
La función acepta cualquier URL, descarga el contenido en memoria con `fetch()` → `arrayBuffer()`, y lo escribe en disco. No hay: (1) validación del Content-Type antes de guardar, (2) límite máximo de tamaño del archivo (un archivo de varios GB agotaría la RAM), (3) validación de que la URL usa HTTPS, (4) timeout de la petición. Esto puede causar un Denial of Service por agotamiento de memoria o disco.

**Justificación:**
CWE-400: Uncontrolled Resource Consumption. CWE-20: Improper Input Validation.

**Prompt para corrección:**
```
En electron/main.js, reemplaza la función `downloadToPath` con la siguiente versión hardened:

const MAX_DOWNLOAD_BYTES = 500 * 1024 * 1024; // 500 MB límite
const DOWNLOAD_TIMEOUT_MS = 60_000; // 60 segundos

const downloadToPath = async (targetUrl, defaultFilename = 'download') => {
  // Validar protocolo
  const parsed = new URL(targetUrl);
  if (parsed.protocol !== 'https:' && parsed.protocol !== 'http:') {
    throw new Error('Protocolo no permitido para descarga');
  }

  const { canceled, filePath } = await dialog.showSaveDialog(mainWindow, {
    defaultPath: path.join(app.getPath('downloads'), sanitizeFilename(defaultFilename, 'download')),
  });
  if (canceled || !filePath) return null;

  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), DOWNLOAD_TIMEOUT_MS);

  try {
    const response = await fetch(targetUrl, { signal: controller.signal });
    if (!response.ok) throw new Error(`Download failed: ${response.status}`);

    const contentLength = parseInt(response.headers.get('content-length') || '0', 10);
    if (contentLength > MAX_DOWNLOAD_BYTES) throw new Error('Archivo demasiado grande');

    const arrayBuffer = await response.arrayBuffer();
    if (arrayBuffer.byteLength > MAX_DOWNLOAD_BYTES) throw new Error('Archivo demasiado grande');

    fs.writeFileSync(filePath, Buffer.from(arrayBuffer));
    return filePath;
  } finally {
    clearTimeout(timeoutId);
  }
};
```

### 🟡 HALLAZGO 008 — MEDIO: `writeJsonFile` sin bloqueo atómico — Posible corrupción de datos

**Ubicación:** `electron/main.js` · Funciones `writeJsonFile()` y `saveState()` (líneas ~60–70, ~450–460)

**Descripción técnica:**
Ambas funciones usan `fs.writeFileSync()` directamente sobre el archivo destino. Si el proceso es interrumpido (crash, kill, apagado del sistema) durante la escritura, el archivo JSON quedará truncado o corrupto, causando errores en el siguiente arranque. Para `passwords.json` esto implicaría pérdida permanente de credenciales almacenadas.

**Justificación:**
CWE-362: Concurrent execution using shared resource. Best practice: atomic writes via temp file + rename.

**Prompt para corrección:**
```
En electron/main.js, reemplaza la función `writeJsonFile` con una versión de escritura atómica:

const writeJsonFile = (filename, value) => {
  const filePath = getJsonPath(filename);
  const tempPath = `${filePath}.tmp`;
  try {
    fs.writeFileSync(tempPath, JSON.stringify(value, null, 2), 'utf-8');
    fs.renameSync(tempPath, filePath);
  } catch (error) {
    // Limpiar archivo temporal en caso de error
    try { fs.unlinkSync(tempPath); } catch {}
    throw error;
  }
};

Aplica el mismo patrón a la función `saveState()`, cambiando fs.writeFileSync 
directo por la escritura atómica a través de un archivo .tmp + rename.
```

Deuda técnica operacional. Riesgo de supply chain inconsistente.

```

   (ya tiene pnpm-lock.yaml y es más eficiente para monorepos).

   git rm package-lock.json
   echo "package-lock.json" >> .gitignore

3. Agrega al package.json:
   "packageManager": "pnpm@9.x.x",
   "engines": {
     "node": ">=20.0.0",
     "pnpm": ">=9.0.0"

4. Actualiza el CI/CD (si existe) para usar pnpm install exclusivamente.
```

### 🟡 HALLAZGO 015 — MEDIO: `src/main.tsx` — Múltiples raíces React sin manejo de error si elemento no existe

**Descripción técnica:**

## RESUMEN EJECUTIVO {#resumen}

### DESCRIPCIÓN DE LA APLICACIÓN

**Blackmind** es un navegador web de escritorio personalizado construido sobre Electron, diseñado específicamente para macOS (con soporte Windows). Actúa como un "shell" minimalista con glassmorphism nativo, sistema de pestañas con React, split-view para ver dos páginas simultáneamente, popup de autenticación OAuth integrado, gestión de bookmarks/contraseñas, y un sistema sofisticado de optimización de memoria. El proyecto exhibe un nivel de ambición arquitectónica considerable para lo que parece ser un browser wrapper de uso personal o empresarial.

---

### ✅ PUNTOS FAVORABLES

1. **Arquitectura modular sólida en Electron**: Los módulos (`CacheCleaner`, `MemoryMonitor`, `FlagsInitializer`, `WindowConfigurator`, `DiagnosticsModule`, `ErrorLogger`) están bien separados y documentados con requisitos numerados.

2. **Seguridad base correcta**: `nodeIntegration: false`, `contextIsolation: true` y el uso de `contextBridge` son las prácticas recomendadas por Electron Security. El `isTrustedOrigin()` centralizado es una buena práctica defensiva.

3. **Cobertura de tests robusta para módulos Electron**: Los tests de `CacheCleaner`, `MemoryMonitor`, y `FlagsInitializer` son exhaustivos, incluyendo tests de propiedad con `fast-check` y pruebas de edge cases.

4. **Manejo de autenticación OAuth bien pensado**: El `authPopupWindow` con `WebContentsView` y la detección por palabras clave es una solución elegante para el problema de login con webviews.

5. **Optimizaciones de rendimiento bien documentadas**: Los flags de Chromium (GPU rasterization, zero-copy, Metal API) están correctamente justificados para el target hardware (Mac M1).

6. **Diseño visual de alta calidad**: El uso de `FloatingLines.tsx` con WebGL/GLSL shaders, `MorphingSquare`, y el sistema de glassmorphism con `vibrancy: 'fullscreen-ui'` refleja un producto con atención al detalle visual.

7. **Escritura en modo atómico de estado**: `saveState` y `readJsonFile` tienen manejo de errores básico con try/catch.
